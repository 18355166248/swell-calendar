# swell-calendar-s2 · P7–P8 后续规划

> 本文件是 S2 外壳应用 P0–P6 全部完成后的下一阶段规划。
> 创建日期：2026-06-12

## 背景

P0–P6 已完成：脚手架搭建、设计稿忠实移植、S2 真组件替换、引擎挂活（scheduler / timeline）、
控件功能化（主题 / 搜索 / 筛选 / 新建 / 编辑删除）、数据源抽象。

当前存在两个主要体验断层：

1. **视图割裂**：scheduler / timeline 走 `packages/calendar` 真引擎（拖拽、resize、车道堆叠），
   但日视图 / 周视图 / 月视图仍是 `views.tsx` 里的静态 CSS 版——不可拖拽、不可滑动新建，
   与引擎事件列表也不共享同一份数据渲染路径。
2. **侧栏纯展示**："我的日历" 分类项和 MiniCalendar 均为静态 markup，无点击交互——
   无法按分类筛选可见事件，也无法通过迷你日历跳转日期。

`packages/calendar` 引擎**已经支持**日 / 周 / 月 / scheduler / timeline 五种视图，
且 scheduler 的滑动新建（`dragToCreate`）能力已内置，只是 S2 外壳尚未接通回调。
因此后续工作量集中在「接线」而非「造轮子」。

## P7 · 全视图引擎化 + 交互闭环

> 让日 / 周 / 月视图也走 `packages/calendar` 真引擎，并把 scheduler 滑动新建接通。
> 完成后所有五种视图共享同一份引擎渲染 + 同一份数据管线。

### 7a · 日视图 / 周视图 / 月视图接入引擎

**现状**：`App.tsx` 的 `useEngine` 只覆盖 `scheduler | timeline`；日 / 周 / 月走静态 `DayView` / `WeekView` / `MonthView`。

**目标**：把三种视图也交给 `<Calendar>` 渲染。

**改动范围**：

- `App.tsx`：
  - `useEngine` 扩展为全视图（移除 `useEngine` 条件，所有视图统一走 `<Calendar>`）。
  - `options` 按视图类型组装：day 走 `defaultView: 'day'`（**无独立 `options.day`**，日视图是「1 列的 week」，复用 `WeekOptions`——`hourStart` / `hourEnd` / `workweek` 均从 `options.week` 读取），week 走 `defaultView: 'week'` + `workweek`，month 走 `defaultView: 'month'`。
  - 移除 `<DayView>` / `<WeekView>` / `<MonthView>` 的条件渲染分支。
  - 所有视图的 `onEventClick` 统一走 `toPickEvent`（已有）。
- `calendarData.ts`：
  - day / week 视图的事件需要正确的 `Date` 对象（当前 `toCalendarEvents` 已经做了，无额外改动）。
- `views.tsx`：
  - 标记 `DayView` / `WeekView` / `MonthView` 为 deprecated 或直接移除（取决于是否还有 Storybook 使用方）。

**风险与注意**：

- **月视图交互受限**：引擎月视图目前不支持拖拽移动 / resize / 滑动新建（SPEC backlog），但点击事件和日期格渲染是完整的。作为 P7 可接受——先让月视图数据源和渲染与引擎统一，交互增强留后续迭代。
- **日视图的小时范围**：引擎 day 视图显示 0–24h 全天，设计稿 day 视图只显示 8–20h。日视图读取 `options.week.hourStart` / `options.week.hourEnd`（无独立 `day` 选项），现 App 已传 `week: { hourStart: 8, hourEnd: 20 }`，**日视图自动继承，零额外配置**。
- **SubBar 显示范围**：当前 SubBar（工作周开关 + 分类 chips）只在 week / day / scheduler 视图显示。引擎化后月视图也应显示分类 chips（用于 7b 的日历过滤联动）。

**验收**：

- 切换到日 / 周 / 月视图时，渲染来自 `<Calendar>` 而非静态组件。
- 事件数据、搜索过滤、分类过滤在五种视图中行为一致。
- `tsc --noEmit` / `vite build` / `pnpm check` 通过。

### 7b · Scheduler 滑动新建接通

**现状**：引擎的 `dragToCreate` 能力已内置（`useGridSelection` → `createEventFromTimeGridSelection`），但 S2 外壳的 `callbacks` 只接了 `onEventClick`，`onEventCreate` 未接通——用户在 scheduler 空白处滑动后什么都不会发生。

**目标**：接通 `onEventCreate` 回调，让滑动选区结束后自动弹出新建对话框并预填时间 / 资源。

**改动范围**：

- `App.tsx`：
  - `callbacks` 新增 `onEventCreate`：从回调参数 `{ event }` 中提取 `start` / `end` / `resourceId`，转成 `EventDraft` 预填信息，然后打开 `CreateDialog`（复用现有新建对话框，增加 `initial` 预填值）。
  - `callbacks` 新增 `onEventUpdate`：把引擎的拖拽移动 / resize 结果转写回数据源（`updateEvent`）。
  - `callbacks` 新增 `onEventCreateFailed` / `onEventUpdateFailed`：可选，显示 toast 或忽略。
- `calendarData.ts`：
  - 新增 `engineEventToDraft(event: EventObject): EventDraft` 辅助函数，从引擎事件对象提取新建 / 更新所需的字段。
  - **更新场景必须基于原事件合并**：`toCalendarEvents` 已将原始 `CalEvent` 挂在 `event.raw` 上。`engineEventToDraft` 在更新路径下必须以 `{...(event.raw as CalEvent), day, start, end, res}` 的方式构建 draft——以原事件为底，只覆盖时间 / 资源。否则 `updateEvent` 的全量 override 语义会把 `title / cat / who / desc` 抹掉。新建路径无 `raw`，从引擎 event 字段直接构建即可。
- `overlays.tsx`：
  - `CreateDialog` 的 `initial` prop 已支持预填，无需额外改动（只需确认 `resourceId` → `res` 映射正确）。

**风险与注意**：

- **onEventUpdate 的日期格式**：引擎回调的 `event.start` / `event.end` 是 `Date` 对象，需要转回 `CalEvent.day`（dayIndex）+ 十进制小时。复用 `dateToDayIndex` / `timeToDecimalHour` 即可。
- **override 语义 + raw 合并**：`updateEvent(id, draft)` 内部是 override 全量替换（P5 语义）。**必须基于 `event.raw` 合并**（见上方 calendarData.ts 说明），否则拖拽只回写 time/resource，会丢失 title/cat/who/desc。
- **onRangeSelect 与 onEventCreate 会同时触发**：滑动选区在引擎内部**同时**触发 `onRangeSelect` 和 `onEventCreate`（`TimeGridView.tsx:222/232`）。外壳**只接 `onEventCreate` 即可**，不要两个都接导致双弹窗。
- **异步数据源延迟 × 拖拽回弹闪烁**：引擎是宿主受控、不维护乐观状态（PLAN4 §4.1），渲染只认 `props.events`。P6 的 `dataSource.ts` 每个写操作有 `180ms` 模拟 IO 延迟：drop → `onEventUpdate` → `updateEvent` → 180ms → 重拉 → 重渲。这中间事件会**先弹回原位、约 200ms 后才跳到新位**。解法二选一：① mutation 路径把 `IO_DELAY` 设为 0（只在首屏 `list()` 保留 loading 态）；② 在 `useCalendarData` 里做乐观更新（mutation 后立即 patch 本地事件列表，重拉后静默替换）。推荐方案 ①，工作量最小。

**验收**：

- 在 scheduler 空白区域按住鼠标拖出一段选区，松开后弹出新建对话框，时间和资源已预填。
- 拖拽事件到另一个资源列或不同时间段，事件位置更新并持久化。
- 拖拽事件底部边缘 resize 调整结束时间，更新持久化。

### 7c · Day / Week 视图引擎交互接通

**目标**：日视图和周视图也能滑动新建和拖拽移动事件。

**改动范围**：

- 与 7b 共用同一套回调（`onEventCreate` / `onEventUpdate`）。
- Day / Week 视图无 `resourceId`，新建时默认资源为空或使用第一个资源。
- 日视图无独立 options 键，小时范围复用 `options.week.hourStart` / `options.week.hourEnd`（已配置为 8–20，**零额外工作**）。

**验收**：

- 在日 / 周视图空白时段滑动新建，弹出对话框预填正确日期和时间段。
- 在日 / 周视图拖拽已有事件移动到其他时段，更新持久化。

## P8 · 侧栏交互完善

> 让左侧 "我的日历" 和 MiniCalendar 从纯展示变为可操作。

### 8a · "我的日历" 分类可见性切换

**现状**：`shell.tsx` 的 `navCal` 列表渲染四个分类（会议·评审 / 规划·设计 / 客户·对外 / 工程·协作），每个是一个 `<button>` 但没有 `onClick`——不可点击、不可切换。

**目标**：点击分类项可 toggle 该分类下事件的可见性（视觉反馈：关闭时变灰 / 划掉），与顶栏 SubBar 的分类 chips 联动。

**改动范围**：

- `App.tsx`：
  - 新增 `activeCalendars: Set<Cat>` 状态（初始全选），与 `activeCats`（SubBar chips）共用同一份集合或合并为单一状态。
  - 传入 `Sidebar` 的 props 增加 `activeCalendars` + `onToggleCalendar(cat)`。
- `shell.tsx`：
  - `SidebarProps` 新增 `activeCalendars: Set<Cat>` + `onToggleCalendar: (c: Cat) => void`。
  - `navCal` 按钮加 `onClick={() => onToggleCalendar(n.c)}`。
  - 视觉态：未选中时按钮加 `opacity: 0.4` + 色块变灰，文字加 `text-decoration: line-through`。
- 事件过滤：
  - `visibleEvents` 的过滤逻辑已经用 `activeCats` 做了分类过滤（P5 搜索过滤切片），只需确保侧栏 toggle 与 chips 操作的是同一份状态。

**风险与注意**：

- **状态一致性**：SubBar chips 和侧栏 "我的日历" 必须反映同一份 filter 状态。如果用户在侧栏关闭了 "会议·评审"，SubBar 对应 chip 也应同步取消选中（反之亦然）。建议合并为一个 `activeCats` 状态，两处 UI 都消费它。
- **侧栏与 SubBar 数量不对称**：侧栏 `navCal` 渲染 4 个分类（会议·评审 / 规划·设计 / 客户·对外 / 工程·协作），而 SubBar 的 `FILTER_CATS` 是 5 个（多一个「面试·招聘」）。合并时需确认 `navCal` 的分类键与 `FILTER_CATS` 对齐——要么侧栏补上第 5 项，要么接受「侧栏关不掉面试·招聘」的不对称。

**验收**：

- 点击侧栏 "会议·评审"，该项变灰，scheduler / 日 / 周 / 月视图中该分类的事件全部隐藏。
- 再次点击恢复显示。
- SubBar chips 与侧栏 toggle 完全同步。

### 8b · MiniCalendar 交互

**现状**：`shell.tsx` 的 `MiniCalendar` 是纯静态组件——月份标题硬编码为 "2025年3月"，左右箭头按钮无 onClick，日期格不可点击。

**目标**：

1. 月份切换：点击左右箭头翻月，标题跟随变化。
2. 日期选择：点击某个日期格，主视图跳转到该日期（`calRef.current.setDate()`）。
3. 今天标记：高亮当前选中日期 / 主视图正在展示的日期。

**改动范围**：

- `shell.tsx`：
  - `MiniCalendar` 改为受控组件：props 接收 `currentDate: Date` + `onDateChange: (d: Date) => void`。
  - 内部新增 `displayMonth` 状态（`useState(() => startOfMonth(currentDate))`），左右箭头改变 `displayMonth`。
  - 日期格点击回调 `onDateChange`。
  - 高亮逻辑：选中日期 + 所在周（沿用现有 `inWeek` / `today` 类名）。
- `App.tsx`：
  - 新增 `currentDate` 状态（初始 `2025-03-21`，与引擎 `initialDate` 一致）。
  - 当 `currentDate` 变化时，调用 `calRef.current.setDate(currentDate)` 同步引擎日期。
  - 当引擎 `onPageChange` 回调触发时（用户通过顶栏箭头翻页），反向同步 `currentDate`，让 MiniCalendar 跟着翻月。
  - **`canvas key={view}` 重挂载冲突**：现有 `<div className="canvas" key={view}>` 在切换视图时会整体重挂载 `<Calendar>`，引擎日期被重置回 `initialDate: '2025-03-21'`，刚导航的日期就丢了。解法二选一：① 把 `currentDate` 格式化后动态传入 `initialDate`（`initialDate={fmt(currentDate)}`），这样重挂载也落在当前日期；② 去掉 `key={view}`，改纯 `setView` 驱动（引擎内部 `ViewRouter` 已支持视图切换无需重挂载）。推荐 ②——避免无谓的重挂载开销，同时 ① 作为兜底保留。
- `Topbar`：
  - 顶栏 "今天" 按钮接通 `calRef.current.goToToday()` + 更新 `currentDate`。
  - 顶栏左右箭头接通 `calRef.current.navigate('prev'|'next')`。

**风险与注意**：

- **日期双向同步**：主视图翻日 ↔ MiniCalendar 翻月需要双向联动。引擎 `onPageChange` 回调可拿到当前日期，用于反向更新 MiniCalendar。MiniCalendar 点击日期时正向调用 `calRef.current.setDate()`。两个方向不能形成循环——`setDate` 触发 `onPageChange`，`onPageChange` 更新 state，state 更新不触发新的 `setDate`（通过 `useEffect` 依赖比较或 ref flag 防抖）。
- **月份计算**：需用 dayjs 或原生 `Date` 做月份加减，不引入新依赖（dayjs 已是 peer dep）。
- **固定单周 mock × 日期导航 = 跨周空白**：`data.ts` 的 `CalEvent.day` 是 0–6 的「周内索引」，`calendarData` 的 `makeDate` 把所有事件锚定在 `2025-03-17`（周一）那一周。后果：MiniCalendar 跳到 4 月 → 主视图能跳，但**没有任何事件**；月视图（2025-03）只有 17–23 那几格有事件。这不是 bug 但属于 UX 死角。**本切片显式 scope**：demo 仅 3/17 那周有数据，跨周 / 跨月导航为空属预期行为。如需跨周有数据，应把 mock 升级为绝对日期（回到 P6 真数据源的延伸工作），不在本切片范围。

**验收**：

- MiniCalendar 左右箭头可翻月，标题正确变化。
- 点击 MiniCalendar 某个日期，主视图跳转到该日（scheduler 显示该日、日视图显示该日、周视图跳到该周）。
- 主视图顶栏翻页后，MiniCalendar 高亮跟随。
- 点 "今天" 按钮回到当前日期。

## 优先级建议

| 阶段 | 切片 | 预估工作量 | 优先级 | 说明 |
|------|------|-----------|--------|------|
| P7 | 7a 全视图引擎化 | 中 | 高 | 消除最大的体验割裂（静态 vs 引擎） |
| P7 | 7b scheduler 滑动新建 | 小 | 高 | 引擎能力已就绪，只差回调接线 |
| P7 | 7c day/week 交互接通 | 小 | 中 | 复用 7b 回调，增量工作很少 |
| P8 | 8a 日历分类切换 | 小 | 中 | 与现有 chips 状态合并即可 |
| P8 | 8b MiniCalendar | 中 | 低 | 日期双向同步有一定调试量 |

建议执行顺序：**7b → 7a → 7c → 8a → 8b**。7b 最小最快（只加回调），验证引擎接线通畅后再做 7a 全视图切换。

## 依赖与前置

- 所有 P7 工作依赖 `packages/calendar` 当前能力（已确认五种视图 + dragToCreate 均就绪），**无前置阻塞**。
- P8 不依赖 P7，理论上可并行推进；但 8b 的日期同步在引擎化视图下更自然，建议 P7 先完成。
- **8b 的 scope 受限于 mock 数据模型**：`CalEvent.day` 是周内索引（0–6），所有事件锚定在 2025-03-17 那周。MiniCalendar 可翻页可跳转，但跨周后视图为空。如需跨周有数据，需将 mock 升级为绝对日期或接入真后端——这是 P6 数据源抽象的后续延伸，不在本规划范围。
- 月视图的拖拽 / resize 交互增强在引擎 SPEC backlog 中，不在本规划范围内——等引擎侧交付后再补。
